# Technical Specification: Payment Registry System (Pilot)

## 1. Data Schema and Entities

### 1.1. Organization (Юридическое лицо)
* **ID**: Primary Key (UUID/Int).
* **Name**: Название (например, Метком-инвест, Айрон, РТПЗ).
* **PaymentGroup**: Группа платежных дней (Group_A: Mon/Wed; Group_B: Tue/Thu).
* **MorningBalance**: Текущий остаток на утро (обновляется Казначеем ежедневно).
* **Conclusion**: This entity serves as the root for fiscal partitioning and liquidity calculations.

### 1.2. Direction (Направление/ЦФО)
* **ID**: Primary Key.
* **ParentID**: Ссылка на родителя (Self-referencing для построения дерева).
* **Name**: Название подразделения (ИТ, Логистика и др.).
* **Conclusion**: Directions must support recursive tree traversal for RLS (Row-Level Security) access.

### 1.3. Payment Request (Заявка на платеж)
* **ID**: Primary Key.
* **InitiatorID**: Автор (сотрудник).
* **OrganizationID**: Ссылка на организацию.
* **DirectionID**: Ссылка на направление.
* **Counterparty**: Контрагент (текстовое поле для пилота).
* **ContractDetails**: Номер счета/договора (текстовое поле).
* **Amount**: Сумма платежа (Decimal 18,2).
* **Purpose**: Назначение платежа (текст для платежного поручения).
* **BudgetItemID**: Статья ДДС (заполняется сотрудником ФЭО).
* **ContractStatus**: Флаг наличия договора (устанавливается Бухгалтерией).
* **DesiredDate**: Желаемая дата от инициатора.
* **PlannedDate**: Утвержденная дата в реестре.
* **Status**: Текущее состояние (Draft, Review, To_Pay, Paid, On_Clarification, Suspended).
* **PriorityFlag**: Признак «Спец. распоряжение».
* **FileStoragePath**: Ссылка на загруженный скан (формат: Контрагент_№Счета).
* **Conclusion**: The Payment Request object is the primary unit of the workflow and must maintain a strict state machine.

### 1.4. Digital Memo (Электронная СЗ)
* **RequestID**: Связь с заявкой.
* **ViolationType**: Тип (После 11:00, Неплатежный день, Вне бюджета).
* **Reason**: Обоснование от инициатора.
* **ApprovalStatus**: Состояние согласования Директором.
* **Conclusion**: Digital Memos replace physical paper signatures and act as a bypass trigger for system constraints.

## 2. Business Logic and System Rules

### 2.1. Time and Calendar Constraints (Gatekeeper)
* **Daily Deadline**: Заявки на текущий день принимаются до 11:00. После 11:00 дата автоматически сдвигается на следующий платежный день.
* **Group Schedules**:
    * Группа А (Айрон, РТПЗ, Метком, ЮТЗ): Платежные дни Пн и Ср.
    * Группа Б (РИК, Вертикаль и др.): Платежные дни Вт и Чт.
* **Friday Restrictions**: В пятницу разрешены только Налоги, ЗП, Банковские платежи и неотложный Транспорт[cite: 55]. Прочие статьи требуют подтвержденной СЗ.
* **Payday Rules (10th/25th)**: Приоритет выплат ЗП и налогов. Поставщики оплачиваются по остаточному принципу.
* **Conclusion**: Logic must enforce "Hard Deny" for dates unless an approved Digital Memo is present.

### 2.2. Workflow and Roles
* **Initiator**: Создает заявку, прикрепляет файл, видит только свои записи.
* **Accounting**: Проверяет наличие договора в реестре в окне с 11:00 до 12:00.
* **FEO (OEF)**: Проставляет статьи ДДС и контролирует бюджетные лимиты.
* **Cashier**: Работает с реестром после 12:00, исполняет платежи до 16:00, проставляет статус «Оплачено».
* **Director/FinDir**: Согласовывает экстренные случаи и внебюджетные расходы.
* **Conclusion**: Access control is implemented via RLS (Row-Level Security) based on OrganizationID and DirectionID.

### 2.3. Budgetary Dialogue
* При выявлении перерасхода ФЭО переводит заявку в статус «On_Clarification».
* Инициатор получает уведомление и обязан либо отозвать платеж, либо инициировать электронную СЗ на перерасход.
* **Conclusion**: System prevents status transition to "To_Pay" if budget violation is detected without an active Memo.

## 3. Interface and UI Requirements

### 3.1. Main Registry (Tree-Grid View)
* Группировка: Организация -> Направление -> Заявки.
* Inline-editing для полей: Статья ДДС, Статус договора, Плановая дата, Статус оплаты.
* Визуальная подсветка «Спец. распоряжений» и «Экстренных» платежей.

### 3.2. File Operations
* При клике на иконку файла в строке реестра открывается модальное окно с предпросмотром счета (PDF/Image).
* Функция «Открыть в новой вкладке» для детального изучения документа.

### 3.3. Liquidity Dashboard
* Виджет для казначея: Сравнение утреннего остатка (MorningBalance) с общей суммой заявок «К оплате» на сегодня.
* Автоматический расчет дефицита/профицита по каждой организации.

## 4. Audit and Logging
* **Activity Log**: Запись каждого изменения (Кто, Когда, Какое поле, Было, Стало). История хранится бессрочно.
* **Conclusion**: Audit logging is mandatory for all fiscal-related fields to ensure non-repudiation.

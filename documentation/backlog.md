# Backlog проекта

Дата создания: 22.04.2026.

## Как вести backlog

Статусы задач:

- `TODO` — задача запланирована;
- `IN PROGRESS` — задача в работе;
- `DONE` — задача завершена и проверена;
- `BLOCKED` — задача заблокирована внешней причиной.

После каждого завершенного блока нужно:

- обновить статус задач в этом файле;
- кратко отметить результат в `documentation/status.md`;
- свериться с владельцем процесса перед переходом к следующему блоку.

## P0 — Regression safety

Статус: `DONE`

Подход: берем весь блок тестов одним проходом — API, Playwright UI и команды запуска. После завершения всего P0 сверяем результаты и только потом переходим к UX.

Цель: закрепить текущий workflow автотестами, чтобы следующие изменения UX и OCR не ломали движение заявок.

### P0.1 API-тесты `gate_preview`

Статус: `DONE`

Критерии готовности:

- покрыт `POST /requests/gate_preview`;
- проверены типы дней `PAYMENT`, `NON_PAYMENT`, `HOLIDAY`, `SALARY_DAY`;
- проверена матрица ДДС: разрешенная категория проходит, запрещенная требует исключения;
- проверено совпадение результата `gate_preview` с фактическим `submit`;
- тесты запускаются понятной командой и описаны в `documentation/testing.md`.

Результат: добавлены Playwright API-тесты `frontend/tests/api/gate-preview.spec.ts`; проверены `gate_preview`, матрица ДДС, совпадение preview/submit и основные API-переходы workflow.

### P0.2 Playwright UI regression

Статус: `DONE`

Критерии готовности:

- создание заявки и preview в форме;
- `DRAFT -> PENDING`;
- `DRAFT -> PENDING_GATE -> Разрешить исключение -> PENDING`;
- `DRAFT -> PENDING_GATE -> Отклонить исключение -> REJECTED`;
- `PENDING -> APPROVED -> PAID`;
- `PENDING -> CLARIFICATION -> повторная отправка`;
- `PENDING -> POSTPONED -> DRAFT`;
- `PENDING -> MEMO_REQUIRED -> Обосновать -> PENDING_MEMO -> директор approve/reject`;
- `PENDING -> MEMO_REQUIRED -> Отменить / Вернуть в черновик`;
- `APPROVED -> SUSPENDED -> PENDING`;
- проверка видимости действий по ролям.

Результат: добавлены Playwright UI-тесты `frontend/tests/ui/workflow.spec.ts`; проверены preview в форме, видимость основных действий по ролям и оплата через Popconfirm без `force`.

### P0.3 Команды запуска тестов

Статус: `DONE`

Критерии готовности:

- добавлены понятные npm/python команды для regression suite;
- порядок запуска описан в `documentation/testing.md`;
- локальный прогон не требует ручных действий, кроме запущенного окружения;
- `npm run build`, API-тесты и Playwright suite проходят.

Результат: добавлены npm-команды `test:api`, `test:e2e`, `test:regression`, `test:regression:headed`. После обновления workflow вне бюджета финальный локальный прогон `npm run test:regression` прошёл: 12/12.

## P1 — UX workflow

Статус: `IN PROGRESS`

Примечание: UX будем обсуждать отдельно после завершения P0. До этого блока UX-задачи не детализируем и не берем в работу.

Цель: сделать движение заявки понятнее для пользователей после стабилизации тестовой сетки.

### P1.1 Карточка заявки

Статус: `IN PROGRESS`

Критерии готовности:

- в карточке явно видны текущий статус и следующий ответственный;
- видны причина исключения, внебюджетное решение, причина отклонения/уточнения/переноса;
- доступны понятные действия по текущей роли;
- история фиксирует ключевые переходы, включая согласование и оплату;
- карточка заявки перевёрстана в горизонтальное представление, чтобы основные данные читались без длинной прокрутки;
- история заявки вынесена на отдельную вкладку и читается как timeline процесса.

Текущий результат: добавлен общий компонент просмотра заявки для основного реестра и рабочего места казначея; форма создания/редактирования заявки перевёрстана в горизонтальные блоки. Данные разложены по блокам, текущие статусы и следующий ответственный видны в верхней части, доступные действия переиспользуют существующую ролевую логику, история вынесена на вкладку `История`; согласование и оплата пишутся в историю. Закрываем после визуального подтверждения.

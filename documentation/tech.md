# Технические требования: Данные и логика (Custom Development)

## 1. Справочники и структура данных

### 1.1. Организации и Направления (Иерархия)
* **Organizations:** Список юридических лиц (плательщиков). Каждое лицо привязано к своему платежному календарю.
* **Directions:** Древовидный справочник подразделений (ИТ, Снабжение, Бухгалтерия и т.д.).
* **Mapping:** Каждое подразделение связано с конкретным Инициатором для автоматизации заполнения заявок.

### 1.2. Статусная модель заявки
* **Draft (Черновик):** Виден только автору.
* **On Review (На проверке):** Виден бухгалтерии (договор) и ФЭО (аналитика).
* **Clarification (Уточнение):** Заявка возвращена инициатору при превышении лимитов.
* **Approved (К оплате):** Полностью проверенная и утвержденная заявка.
* **Suspended (Пауза):** Замороженные платежи (бэклог).
* **Paid (Оплачено):** Финальный статус после проведения выписки.

## 2. Алгоритмы «Умного шлюза»

### 2.1. Контроль временных окон
* Автоматическая установка даты на следующий рабочий день при подаче после 11:00.
* Механизм «Экстренного запроса» (Override): Поле текстового обоснования, которое открывает возможность оплаты «день в день» только после акцепта ФЭО.

### 2.2. Контроль платежных дней
* Группировка организаций по дням недели (Пн/Ср и Вт/Чт). 
* Проверка соответствия выбранной даты календарю группы при сохранении заявки.
* Блокировка стандартных расходов в Пятницу (исключения: Налоги, ЗП, Банк, Транспорт).

### 2.3. Электронное согласование СЗ
* Вместо прикрепления сканов используется встроенная форма «Обоснование исключения».
* Статус «Согласовано Директором» — обязательный триггер для снятия блокировки с заявок, поданных вне бюджета или в неплатежный день.

## 3. Интерфейс и UX (Frontend)

### 3.1. Реестр (Tree-Grid)
* Группировка строк по Организации (первый уровень) и Направлению (второй уровень).
* Inline-редактирование полей: Статья ДДС, Статус договора, Плановая дата.
* Модальное окно просмотра счета (PDF/Image) при клике на иконку файла в строке.

### 3.2. Казначейский модуль
* Экран ввода утренних остатков по расчетным счетам.
* Индикатор ликвидности: Автоматический расчет разницы между остатком на утро и суммой утвержденных заявок на сегодня.

## 4. Системные функции
* **Audit Log:** Полное логирование всех действий (ID пользователя, время, поле, старое значение, новое значение).
* **Notification Engine:** Рассылка уведомлений инициаторам об оплате, переносе или возврате заявки на уточнение.

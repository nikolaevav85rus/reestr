# Техническая документация Backend: Реестр Платежей

## 📌 Общий обзор
Система представляет собой асинхронный API для управления корпоративными платежами. Центральным узлом является **Smart Gateway (Умный шлюз)**, который в реальном времени валидирует заявки на соответствие финансовым регламентам компании.

**Стек технологий:**
* **FastAPI**: Веб-фреймворк.
* **SQLAlchemy (Async)**: ORM для работы с БД.
* **asyncpg**: Драйвер PostgreSQL для высокой производительности.
* **Alembic**: Управление миграциями базы данных.

---

## 🏗 Архитектура данных (ER-модель)

Система построена на пяти логических блоках:

1.  **Организационная структура**: `PaymentGroup` (холдинги) и `Organization` (юрлица).
2.  **Финансовая структура**: `Direction` (ЦФО) и `BudgetItem` (Статьи ДДС с категориями).
3.  **Механизм правил**: `DayTypeRule` (Матрица соответствия типов дней и категорий ДДС).
4.  **Календарный движок**: `WeeklyTemplate` (шаблоны недели) и `PaymentCalendar` (фактические даты).
5.  **Процессинг заявок**: `PaymentRequest` (основная сущность) и `AuditLog` (история всех изменений).

---

## 🧠 Логика «Умного шлюза»

Шлюз срабатывает при создании заявки (`POST /requests/`) и проверяет:

* **Time Check**: Если платеж запланирован на "сегодня", время сервера должно быть меньше **11:00 МСК**.
* **Calendar Check**: Дата платежа должна существовать в календаре и не быть помечена как `HOLIDAY`.
* **Matrix Check**: Категория выбранной статьи ДДС должна быть разрешена для типа дня, указанного в календаре для данной организации.

### Режим спец. распоряжений (Special Order)
Если шлюз обнаруживает нарушение правил, он блокирует заявку (`400 Bad Request`). Однако, если в запросе передан флаг `special_order: true`, заявка создается, но:
1.  Автоматически получает статус **PENDING** (на согласовании).
2.  В поле **feo_note** записывается причина нарушения.
3.  Факт нарушения фиксируется в **AuditLog**.

## 🚦 Статусная модель

Разделение ответственности за согласование и оплату:

* **ApprovalStatus**: `DRAFT` → `PENDING` → `APPROVED` / `REJECTED` (или `POSTPONED` для переноса).
* **PaymentStatus**: `UNPAID` → `PAID` (фиксация факта списания средств).

## 🛠 Карта API Эндпоинтов

### 🔑 Авторизация и Пользователи
* `POST /auth/login` — получение JWT токена.
* `POST /users/` — регистрация нового пользователя.
* `GET /users/` — список всех пользователей.

### 💰 Заявки (Requests)
* `POST /requests/` — создание заявки (включает логику Шлюза).
* `GET /requests/my` — список заявок текущего пользователя.
* `GET /requests/all` — сводный реестр для ФЭО и Бухгалтерии (поддерживает фильтрацию по статусам).
* `POST /requests/{id}/approve` — утверждение заявки (роль: FEO).
* `POST /requests/{id}/pay` — отметка об оплате (роль: CASHIER/ADMIN).

### 📚 Справочники (Dictionaries)
* `GET/POST /dict/payment_groups` — управление платежными группами.
* `GET/POST /dict/organizations` — управление списком юридических лиц.
* `GET/POST /dict/directions` — справочник ЦФО (Направлений).
* `GET/POST /dict/budget_items` — справочник статей ДДС.

### 📅 Календарь и Правила (Calendar)
* `GET/POST /calendar/rules` — настройка матрицы соответствия типов дней и категорий.
* `POST /calendar/generate` — автоматическая генерация календарных дней на месяц для группы на основе её шаблонов.

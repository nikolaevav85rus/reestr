# План ручного тестирования

Дата актуализации: 22.04.2026.

Цель: полный ручной прогон функционала реестра платежных заявок без заполнения НСИ. Справочники, календари, пользователи, роли и тестовые файлы считаются уже подготовленными.

## Окружение

- Frontend: `http://localhost:5173`
- Backend: `http://127.0.0.1:8080`
- Тестовые пользователи:
  - `admin1` / `1234` — администратор
  - `initiator1` / `1234` — инициатор
  - `feo1` / `1234` — ФЭО
  - `cashier1` / `1234` — казначей
  - `accountant1` / `1234` — бухгалтер
  - `director1` / `1234` — директор

## Правила прогона

- Не заполнять и не менять НСИ в рамках этого плана.
- Все заявки создавать через UI.
- Для каждой тестовой заявки добавлять уникальный маркер в описание или назначение платежа: `MANUAL-YYYYMMDD-NN`.
- Использовать разные уже существующие организации, ЦФО, контрагентов, статьи ДДС, суммы и даты оплаты.
- Файлы счетов брать из подготовленной папки с типовыми счетами.
- Тестовые заявки не удалять после прогона, если отдельно не указано обратное.
- Статусы в плане указаны так, как они должны отображаться в UI: `Черновик`, `Требует исключения`, `На согласовании`, `Требует обоснования`, `Вне бюджета`, `Ожидание согласования`, `Согласовано`, `Отклонено`, `На уточнении`, `Перенесено`, `Отложена`, `Не оплачено`, `Оплачено`.

| RIGHTS-09 | Директор | Утвердить/отклонить вне бюджета | Доступно только для заявок `Вне бюджета` |
| RIGHTS-10 | Администратор | Просмотр и административные действия | Доступны согласно роли администратора |

## 17. Что фиксировать в отчете

Для каждого сценария фиксировать:

| Поле | Что писать |
|---|---|
| ID | Идентификатор сценария из плана |
| Роль | Пользователь, под которым выполнялась проверка |
| Маркер заявки | `MANUAL-YYYYMMDD-NN` |
| Входные данные | Организация, ЦФО, статья ДДС, дата, сумма, файл |
| Ожидание | Ожидаемый статус/действие |
| Факт | Что реально произошло |
| Результат | `PASS`, `FAIL`, `BLOCKED` |
| Баг | Ссылка или краткое описание, если найден дефект |

## 18. Финальная проверка после ручного прогона

- Все созданные заявки находятся по маркерам `MANUAL-*`.
- В истории заявок есть ключевые события: отправка, согласование, оплата, отклонение, уточнение, перенос, обоснование вне бюджета, отложение.
- В UI используются только бизнесовые термины: `исключение`, `отложена`, `вне бюджета`, `требует обоснования`.
- В колонке `Согласование` нет управляющего select, только статус.
- Основной реестр и рабочее пространство казначея открываются без заметных задержек.
- После ручного прогона желательно отдельно выполнить автоматический контроль:
  - `npm run test:api`
  - `npm run test:regression`

## 1. Технологический стек

Проект реализован в рамках клиент-серверной архитектуры (Single Page Application + REST API).

### Фронтенд (Клиентская часть)
* **Библиотека UI:** React (с использованием функциональных компонентов и хуков).
* **Язык программирования:** TypeScript (строгая типизация интерфейсов и API-ответов).
* **UI-фреймворк:** Ant Design v5 (использование современных компонентов: `Table`, `Modal`, `Form`, контекстный хук `App.useApp` для уведомлений).
* **Роутинг:** React Router v6 (навигация, защита маршрутов через `ProtectedRoute`).
* **Управление состоянием:** Zustand (глобальный стор авторизации `authStore`).
* **Сетевые запросы:** Axios (клиент `apiClient` с интерцепторами для обработки токенов).

### Бэкенд (Серверная часть)
* **Фреймворк:** FastAPI (асинхронный REST API).
* **Язык программирования:** Python 3.10+.
* **ORM:** SQLAlchemy 2.0 (асинхронный движок).
* **База данных:** PostgreSQL (через асинхронный драйвер `asyncpg`).
* **Миграции БД:** Alembic.
* **Валидация данных:** Pydantic.
* **Аутентификация/Безопасность:** JWT (JSON Web Tokens), хэширование паролей.

## 3. Базовая бизнес-логика и паттерны

### Ролевая модель доступа (RBAC)
В системе реализована динамическая матрица прав. Каждому пользователю назначается одна Роль. Роль содержит набор конкретных разрешений (пермиссий), сгруппированных по категориям (например: `dict_edit`, `dict_delete`, `user_view`, `user_edit`).
* **Фронтенд:** Компонент `<HasPermission>` скрывает UI-элементы (кнопки, вкладки), если у текущего пользователя нет нужного ключа.
* **Бэкенд:** Зависимость `Depends(PermissionChecker("..."))` блокирует выполнение эндпоинта на уровне API, возвращая `HTTP 403 Forbidden`.

### Row-Level Security (Ограничение видимости данных)
Заложена основа для RLS. В модуле платежей обычный сотрудник (Инициатор) сможет получать и просматривать только те заявки, которые создал он сам или которые относятся к его ЦФО, в то время как пользователи с расширенными правами или флагом `is_superadmin` смогут видеть общий реестр.

### Контроль целостности данных (Dependency Checking)
На уровне API реализована жесткая защита от «осиротения» данных:
* При попытке удалить справочный элемент выполняется проверка зависимостей.
* *Мягкая проверка (Pre-check):* Запросы `select().limit(1)` перед удалением (например, нельзя удалить ЦФО, если в нем числятся сотрудники). Возвращает `HTTP 400` с детализированным текстом.
* *Жесткая проверка (Database constraints):* Отлов исключений `IntegrityError` (нарушение внешних ключей) с выполнением `db.rollback()` и трансляцией в безопасное `HTTP 400` сообщение для интерфейса.

### Оптимизация UX/UI
* **Локальная фильтрация:** Для небольших справочников (до тысячи записей) применяется паттерн "Загрузить всё при монтировании -> Фильтровать на клиенте". Это исключает сетевые задержки при поиске.
* **Предварительный рендеринг:** Использование `forceRender` и `destroyOnHidden` в модальных окнах Ant Design предотвращает ошибки монтирования форм и утечки памяти.
* **Сортировка и фильтрация:** Встроенные фильтры колонок и кастомные формы поиска в заголовках таблиц.

### РЕЕСТР ЗАЯВОК — ДОРАБОТКИ

**~~Блок 5.1: Submit + PENDING_GATE (шлюз и экстренные платежи)~~ ✅ ЗАКРЫТ**

Машина состояний (полная, согласована с инструкцией от 01.06.2024):

```
DRAFT / CLARIFICATION / POSTPONED
  └─► [Инициатор: «Отправить на согласование»]
        ├─ Время ≤ 11:00 И payment_date — платёжный день  → PENDING
        └─ Нарушение шлюза (после 11:00 / нерабочий / зарплатный день)
              → PENDING_GATE  (ФЭО получает мгновенное уведомление)

PENDING_GATE
  └─► [ФЭО: gate_approve]
        ├─ «Разрешить» (с комментарием)
        │     → special_order = True  (автоматически)
        │     → gate_approved_by = id ФЭО
        │     → gate_reason = комментарий ФЭО
        │     → approval_status = PENDING
        └─ «Отклонить» (с причиной)
              → approval_status = REJECTED
              → инициатор получает мгновенное уведомление

PENDING
  └─► [ФЭО: req_approve]
        ├─ Согласовать    → APPROVED
        ├─ Отклонить      → REJECTED       (причина → инициатор уведомлён)
        ├─ На уточнение   → CLARIFICATION  (комментарий → инициатор уведомлён, правит и re-submit)
        └─ Перенести      → POSTPONED      (причина → инициатор уведомлён, меняет дату и re-submit)

APPROVED
  └─► [Специалист по платежам: req_pay / req_suspend]
        ├─ «Оплатить»    → PAID
        └─ «Отложить»   → SUSPENDED  (см. Блок 5.3)

REJECTED
  └─► Финальный статус (инициатор видит причину, может создать новую заявку)
```

Реализовано:
* Миграция `f1g2h3i4j5k6`: поля `gate_approved_by UUID FK users`, `gate_reason TEXT` в `payment_requests`. Применена.
* RBAC: права `gate_approve` (ФЭО) и `req_suspend` (Казначей) созданы в БД и назначены ролям через `seed_rbac_matrix.py`.
* Новые статусы в `ApprovalStatus`: `PENDING_GATE`, `PENDING_MEMO`, `SUSPENDED` (модель хранит строку — расширение без миграции).
* `POST /requests/{id}/submit` — двойной шлюз: время > 11:00 МСК → PENDING_GATE; тип дня в платёжном календаре организации не PAYMENT → PENDING_GATE; иначе → PENDING.
* `POST /requests/{id}/approve_gate` — `gate_approve`; устанавливает `special_order=True`, `gate_approved_by`, `gate_reason`, переводит в PENDING.
* `POST /requests/{id}/reject_gate` — `gate_approve`; переводит в REJECTED с причиной.

### БАГИ

**~~Б-3: 500 при создании заявки — отсутствие relations после flush~~ ✅ ИСПРАВЛЕН (14.04.2026)**
* Симптом: `POST /requests/` возвращал 500 при сериализации ответа через `RequestResponse`.
* Причина: `create_payment_request` делал `db.refresh(new_request)` — без загрузки relations (organization, direction, budget_item, creator). Async SQLAlchemy не поддерживает lazy load.
* Решение: заменить `db.refresh()` на `return await get_request_by_id(db, new_request.id)`, который использует `selectinload`.

**~~Б-4: 500 при логине — `is_superadmin = NULL` в БД~~ ✅ ИСПРАВЛЕН (14.04.2026)**
* Симптом: `POST /auth/login` возвращал 500 для пользователей с обычными ролями.
* Причина: Pydantic `RoleResponse.is_superadmin: bool = False` не принимал `NULL` из БД.
* Решение: изменить тип на `Optional[bool] = False`.

**~~Б-5: `user.is_superadmin` на фронте всегда `undefined`~~ ✅ ИСПРАВЛЕН (14.04.2026)**
* Симптом: admin не получал права суперадмина на фронте — все `canXxx` флаги были `false`.
* Причина: `UserResponse` не передавал `is_superadmin` — поле есть на `Role`, но не сериализовалось в ответ.
* Решение: добавлен `@computed_field is_superadmin` в `UserResponse` (вычисляется из `role.is_superadmin`). Также `RoleResponse` теперь включает `is_superadmin`.

**~~Б-6: ФЭО/Директор/Бухгалтер получали 403 на `/requests/all`~~ ✅ ИСПРАВЛЕН (14.04.2026)**
* Причина: эндпоинт требует `req_view_own`, но роли DIRECTOR и ACCOUNTING не имели этого права в seed. FEO и CASHIER — seed не был перезапущен после Блока 10.
* Решение: добавлен `req_view_own` ролям DIRECTOR и ACCOUNTING в `seed_rbac_matrix.py`. Матрица обновлена в БД напрямую.

**~~Найдено пользователем: SQLAlchemy InvalidRequestError после Блока 5.1~~ ✅ ИСПРАВЛЕН**
* Симптом: фронт показывал «Ошибка при загрузке заявок», в логах — `InvalidRequestError: Could not determine join condition ... relationship PaymentRequest.creator — there are multiple foreign key paths linking the tables`.
* Причина: добавление `gate_approved_by UUID FK users` создало два FK на таблицу `users` в одной модели — SQLAlchemy не смог автоматически определить какой использовать для `relationship("User")`.
* Решение: явно указан `foreign_keys` для обоих relationships в `app/models/request.py`:
  ```python
  creator = relationship("User", foreign_keys=[creator_id])
  gate_approver = relationship("User", foreign_keys=[gate_approved_by])
  ```
